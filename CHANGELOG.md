## Unreleased

- `2026-07-09 | repo | no | explicit-agent-audience`: none of the four primary agent-facing entry points (`CLAUDE.md`/`AGENTS.md`, root and template) explicitly told a human reader they weren't the intended audience — they talked about or to agents, but never explicitly ruled a person out, relying on filename convention alone. Added an explicit "for AI agents" line to each, pointing a human reader to `README.md`/`CONTRIBUTING.md` instead. Marked `docs/agent-security.md` as intentionally dual-audience rather than leaving it ambiguous. Added "Explicit audience" as a design principle in `CONTRIBUTING.md` for future agent-facing files

- `2026-07-09 | conventions | no | separate-detection-from-tier`: a real Arranger session's own explanation of why the check ran conflated "this project mentions agentics" (adoption detection, a property of the project) with "the check is mandatory for me" (contributor status, a property of the agent's global context) — it still reached the right answer, but the phrasing showed the two ideas sit close enough in `convention-levels.md` § Checking for upstream updates to blur together. Reordered the section so detection is answered first and stated as a project property, then tier strength second and stated as an agent property, with an explicit line saying a project mentioning agentics does not by itself make the check mandatory

- `2026-07-09 | conventions | no | verify-before-writing-dev-docs`: propagated from a real project session (Arranger): verifying against actual code/file state before marking a roadmap or tech-debt item done/resolved held up well across a full session, not just once. Added to `conventions/session-discipline.md` and `template/AGENTS.md` § Keeping `.dev/` current: before writing a status change, verify against current reality, not a prior description or session summary

- `2026-07-09 | conventions | no | contributor-check-explicit-override`: the mandatory contributor tier (previous entry) still lost to a project's own complete session-start checklist, the same pattern that blocked the opt-in tier before. Confirmed by rebuilding that exact blocking pattern in a throwaway fixture: a global instruction phrased as "run this every session" was ignored in favour of the project's own explicit checklist; rephrased to state the override directly — name the failure mode (complete local checklist, narrower dispatch line naming other topics) and say neither opts a project out — and it worked. Added this requirement to `conventions/convention-levels.md` § Checking for upstream updates as an explicit warning with the tested wording pattern, since a plain "mandatory" flag reads as a frequency instruction, not a precedence one

- `2026-07-09 | conventions | yes | contributor-mandatory-upstream-check`: split "checking for upstream updates" into two tiers: opt-in (`propagation_suggestions: yes`, existing behaviour, unchanged) and mandatory for agentics contributors (`agentics_contributor: yes`) — for contributors the check now runs every session in every project regardless of that project's own opt-in setting, and keeps recurring for an unresolved gap instead of surfacing once and going quiet. Stops only on an explicit `agentics_upstream_check: no`, recorded per-project ("stop for this project") or in global context ("stop for all projects") — never inferred from silence. `agentics_contributor: yes` itself now needs recording in global context rather than only agentics' own project memory, since it must apply across every project a contributor works in, not just sessions inside agentics. Updated `conventions/convention-levels.md`, `conventions/session-discipline.md`, `template/AGENTS.md`, `CONTRIBUTING.md`, and both root `CLAUDE.md`/`AGENTS.md` initialization blocks. Breaking: contributors who already set `agentics_contributor: yes` only in agentics' project memory need to re-record it in their global context for the mandatory tier to actually apply elsewhere

- `2026-07-09 | conventions | no | restore-unreleased-catchup`: blind-testing the no-tag path against a clean fixture (no legacy checklist blocking it) surfaced that the "surface current `## Unreleased` as a bounded first-encounter catch-up" behavior had been lost when `convention-levels.md` was refactored to defer to `upgrading-adoption.md`: each file pointed at the other for "the surfacing/catch-up treatment," and neither actually contained it anymore.
