Fixed a real bug reported from an adopting project (Usher): every file in its `.dev/sessions/` carried `T000000`, not just the current day's, because the convention named the filename format but never instructed the agent to actually fetch the current time.

- `template/conventions/session-discipline.md`, `template/AGENTS.md`: added "Get the actual time, don't pad with zeros" to § Session file identity, instructing a shell call (`date +%Y-%m-%dT%H%M%S`) before creating a new session file, and naming an all-zero timestamp pattern as the tell that this step was skipped
- `template/conventions/session-discipline.md`, `template/AGENTS.md`: added an exception for migration/backfill (splitting a legacy shared log, reconstructing past work): a real time genuinely isn't recoverable there, so `T000000` stays a legitimate placeholder for that case and isn't something to rename retroactively
- `template/conventions/session-discipline.md`, `template/AGENTS.md`: reordered § Session file identity so the existence check ("is there already a file for you today?") comes before the time-fetch instruction, since the original order could read as "always fetch a time and create a file" rather than "only do that once the check says none exists"
- `template/conventions/session-discipline.md`, `template/AGENTS.md`: added "Never rename an already-created file, today's own included" after a live adopter's agent renamed its own already-existing today's-file to the current time as a "best approximation of session start" once told about this fix: the same fabricated-timestamp anti-pattern, just relabelled, since a file's real creation time can't be recovered after the fact by any rename
- `CHANGELOG.md`: logged `session-timestamp-no-time-source` under Unreleased
