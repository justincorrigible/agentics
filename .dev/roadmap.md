## Phase 3: personal context portability

Goal: a user can maintain a private GitHub repo mirroring their `~/.claude/` global setup (CLAUDE.md customizations + global memories), so that switching to a new Claude account doesn't erase accumulated rapport and preferences.

Design sketch:
- User creates a private repo (e.g. `<user>/claude-context`) following a structure agentics defines
- Suggested structure: `CLAUDE.md` (global instructions snapshot), `memory/MEMORY.md` + memory files, optionally `projects/<project>/memory/` for project-specific memories
- agentics provides: template for the private repo structure, a standard bootstrap prompt the user gives a fresh agent to reconstitute context from the private repo, guidance on what belongs there (preferences, feedback patterns, working style) and what does not (credentials, project-specific ephemera, anything derivable from code)
- Sync discipline: end-of-session or on-demand; the user commits manually: no automated push

Open questions:
- Should agentics include a `template/personal-context/` scaffold, or document the pattern only in prose?
- How does the bootstrap prompt discover which projects to also load memories for?
- Where does the private repo live relative to agentics in the contribution/update-propagation model?

## Design questions: needs more thinking

- **Startup instruction-file integrity check:** at session start, run `git log --oneline -- CLAUDE.md AGENTS.md` and flag unexpected changes not made by the repo's lead developer; useful tamper signal but adds a mechanical step: decide whether this is a hard rule, optional guidance, or a softeng-specific convention
- **Context sync across projects:** the upstream-update check (`convention-levels.md` § Checking for upstream updates) now surfaces drift automatically for opted-in projects, but the "source of truth" ownership model is still informal: which conventions are global, which are template, which are project-specific, and how to reason about overlap when a project has genuinely diverged on purpose

- **Integration with existing agent setups:** the current approach (initialization question + "supplementary" flag) is minimal; deeper questions remain: should the agent offer to review existing conventions for conflicts? How prescriptive should softeng conventions be when a rich setup is already in place? Is "supplement on conflict" even the right default, or should users choose a merge strategy?
- **Non-softeng users:** the general layer already serves anyone, but the template flow and README still implicitly frame softeng as the "extra" and everyone else as the base case; explore whether the initialization flow, framing, and contribution model should be more explicitly welcoming to teams outside softeng adopting the pattern wholesale
- **Dev vs. non-dev users:** the template currently assumes a developer audience (code reviews, test conventions, OWASP); consider a third initialization question: "Do you work primarily in code, or in planning/documentation/research?": and distinct dispatch paths or convention sets for each; non-dev users may benefit from lighter session discipline and different convention files (e.g. writing style, doc structure) rather than testing and code-style
- **Response verbosity/tone calibration per role:** distinct from the item above, which is about which conventions load. This is about how the agent talks once it's using them. A session working through the upstream-update-check design was appropriately thorough for a dev/AI-eng/contributor audience (file diffs, precedence mechanics, why something changed) — the same level of detail would read as noise to a general (non-dev) user, who likely wants outcomes and next steps with detail available on request, not by default. `CLAUDE.roles/*.md` currently shape *what* applies per role, not *how verbose or technical* the agent's own responses should be for that role; needs its own dispatch, probably as a short section per role file rather than a new mechanism. **A second, mode-based axis, independent of role:** noticed directly in a design/planning session, not reported secondhand: a lengthy, comprehensive response during active back-and-forth design work gets partially skimmed and disregarded rather than fully read, and the user responds anyway, continuing the conversation around what the response missed. The response was trying to be thorough, but the actual effect was a longer, more interactive exchange than a shorter, more focused one would have produced, the opposite of what thoroughness was meant to achieve. This applies even within a single role/audience: a dev/contributor session in active design mode still wants short, reply-inviting turns, not comprehensive up-front coverage, the same instinct that overwrites the role-based calibration above rather than composing with it. Unsolved part: reliably detecting "active design/planning conversation" as a mode, distinct from "reporting completed work" or "final review," the same kind of detection problem session-start signals took iteration to get right
- **"Modality" as a general interaction-mode framework:** parked here deliberately, not designed, so it isn't lost. Human collaborators enter different mental modes depending on task, personality, and ability. "Collaboration" (planning, workshopping, brainstorming: back-and-forth, not user-issues/agent-executes) is one instance already being worked out, see the "Response verbosity/tone calibration" item above. Others named so far, not defined: "learning" (varies by learning style, resists generalization), "documenting" (user-heavy collaboration, already observed in practice; this very entry is an instance of it). "Imperative" (bare commands) is not itself illegitimate: a concise instruction can still carry full purpose and intent. The actual failure mode is a command issued with no regard for plan, purpose, or intent at all, which "verify purpose alignment before implementing" (`code-style.md`/`AGENTS.md` Interaction parameters) already exists to catch from the agent's side. Whether "modality" becomes its own detection mechanism, reusing the explicit-phrase-plus-inferred-signal-plus-confirm pattern from collaboration mode, or stays a loose descriptive frame, is undecided
- **Project memory hygiene: pruning locally-superseded items:** the propagation question (`convention-levels.md`) covers where a *new* convention should live, but nothing currently prompts revisiting *existing* project memory once something it contains gets added to global context later. A local item can end up duplicating, or fully superseded by, a global one added after the local item was written, with nothing flagging the redundancy. Needs an occasional pass — distinct from the roadmap/tech-debt staleness pass, which is about correctness, not this — checking project memory for items global now already covers, and removing them
- **Roadmap done/not-done separation:** `session-discipline.md` § Keeping `.dev/` current (and this file's own header) currently has completed roadmap items removed entirely rather than kept anywhere. Consider relocating finished items to a separate "Done" section after the open ones instead of deleting them, so recently-finished work stays visible without cluttering the open list. Ordering (within the done section, and relative to the open one) needs more thought before this is a concrete rule
- **Coding conventions proposed for later discussion, not yet agreed:** a colleague proposed the following for `code-style.md`, applicable to code review, PR review, and code produced from design sessions alike. Not adopted: discuss and decide before adding any of these.
  - No `Type` suffix on type names, no `I` prefix on interfaces: descriptive names only (`ThemeContext`, not `ThemeContextType`); interface and type names look the same either way
  - Declare return types explicitly on named/declared functions; anonymous functions passed as arguments may omit them
  - Full-word, descriptive variable names always, never a single letter or an abbreviation (`props` not `p`, `count` not `i`)
- **Session-completion checklist: tests, in-code docs, `/docs`, and devctx, all reviewed before "done":** before marking a unit of work done, review and update whichever of these are relevant: tests (`testing.md`), in-code documentation including comments where the WHY is non-obvious (`code-style.md`'s Comments and TSDoc rules), `/docs` or equivalent external-facing docs if the project has them (`documentation.md`'s two-tier model), and `.dev/` (`session-discipline.md`'s existing "Keeping `.dev/` current"). Three of these four already have their own convention individually; what's missing is the single completion gate tying them together, so a change doesn't get called done while one of the four quietly lagged behind unnoticed. Needs shaping: a new section, or a cross-reference added to each existing file pointing at the others?
- **Release process convention: `[in progress]` elsewhere, not here:** a comprehensive release-process pattern (versioning, branch cutting, changelog discipline) is being actively abstracted into its own pattern document in Arranger's own project context, not agentics. Don't draft a release-process convention here from secondhand description: wait for that document, then propagate the stable pattern here as a shareable convention, the same "prove it in a real project before templating it" path agentics' own mechanisms (e.g. the upstream-check, tested against fixtures before being written up) already followed
